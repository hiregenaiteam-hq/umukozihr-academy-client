# Tech stack (MVP → production)

Keep MVP minimal and Python-native, then add infra as needed.

MVP:

* **Backend framework:** FastAPI (async, fast, great dev DX)
* **DB:** PostgreSQL (relational, full-text search via pg_trgm)
* **ORM/Migrations:** SQLAlchemy + Alembic
* **Background jobs:** Redis + RQ or Celery (RQ simpler)
* **Cache / sessions:** Redis
* **File/media:** S3-compatible storage (AWS S3 or DigitalOcean Spaces)
* **Auth:** Auth0 / Clerk / or self-hosted Keycloak (Auth0/Clerk = faster) with SSO across subdomains
* **Hosting:** Render / Railway / AWS Fargate (Render/Railway for quick MVP)
* **Observability:** Sentry (errors) + Prometheus/Grafana or Datadog later
* **CDN & DNS:** Cloudflare (caching, security)
* **CI/CD:** GitHub Actions

Production additions later:

* Search index: ElasticSearch or Postgres full-text until scale
* Event stream: Kafka or managed Kinesis if massive analytics loads

---

# Core services (what the backend actually is)

1. **CMS API** — create/edit/publish posts, authors, thumbnails
2. **Auth service** — SSO for all subdomains (single login) + roles (admin/editor/author)
3. **Analytics ingestion** — record events: post_opened, post_scrolled, post_shared, cta_clicked
4. **Admin dashboard API** — aggregated metrics endpoints for internal dashboard
5. **Media service** — signed uploads to S3, image transforms via CDN
6. **Publishing pipeline** — background task for thumbnails, social preview generation, notify subscribers

---

# Minimal DB schema (core tables)

Use Postgres. Keep tables small.

Posts

* id (uuid, pk)
* slug (unique)
* title
* body (markdown / richtext)
* excerpt
* status (draft/pending/published)
* author_id (fk)
* category (enum: hr, talent, team)
* thumbnail_url
* published_at, created_at, updated_at

Authors

* id (uuid)
* name
* bio
* avatar_url
* approved (bool) — vetting flag
* role (author/editor/admin)
* linkedin_url, org

Events (analytics raw)

* id (uuid)
* event_type (post_opened, post_scrolled, post_shared, cta_clicked)
* post_id (nullable)
* author_id (nullable)
* user_id (nullable) — if logged in
* meta jsonb (browser, referrer, scroll_percent, share_target)
* created_at (ts)

Aggregates (materialized)

* post_id, date, views, uniques, avg_time, scroll_50pct, shares, cta_clicks

Audit & Moderation

* submission_queue (if applied authors)
* moderation_logs

# API endpoints (examples)

Auth routes (via provider)

* `POST /auth/callback` — handle SSO tokens (or use middleware)

CMS

* `GET /api/posts` — list (filters by category, status public)
* `GET /api/posts/{slug}` — read post (increment event hook)
* `POST /api/posts` — create (editor/author)
* `PUT /api/posts/{id}` — update
* `POST /api/posts/{id}/publish` — publish (editor)
* `DELETE /api/posts/{id}`

Authors

* `GET /api/authors`
* `GET /api/authors/{id}`
* `POST /api/authors/apply` — author application flow

Analytics

* `POST /api/analytics/event` — record event (frontend sends)
* `GET /api/analytics/post/{post_id}` — aggregated stats (admin-only)

# Analytics flow (simple, reliable)

1. Frontend sets a long-lived `anon_id` cookie (UUID) on first visit.
2. On post open / scroll / share / CTA: batch events and send to `/api/analytics` server route (server uses Supabase service key to insert into `events`).
3. Every hour or daily run an Edge Function to aggregate `events` to `post_aggregates`. Keep raw events 90 days, aggregates forever.

Why server route? prevents users from spamming directly into DB with anon public keys and allows rate-limiting.
